# 個人用RAG＋エージェント環境（OrbStack） システム要件定義書

## 1. プロジェクトの背景と目的

本プロジェクトは、本来エンタープライズ（銀行本番環境）向けに設計された「VPC閉域網・完全マネージドなセキュアRAGアーキテクチャ」のコアコンセプトを維持しつつ、個人開発用のローカル環境に極限まで最適化・ダウンサイジングしたショウケース（デモ環境）を構築することを目的とする。
将来的に社内へ本番環境の構築を提案する際の「動くモックアップ」として機能させると同時に、AIエージェントによる自律的なシステム構築・運用のナレッジを蓄積するための基盤とする。

## 2. システム制約とハードウェア特性

本システムは、以下の厳格なリソース制約の下で設計・稼働しなければならない。

- **ホストマシン**: Mac (M4チップ)
- **メモリ制約**: 16GB ユニファイドメモリ
  - *設計方針*: 16GBという限られたメモリ内で、開発用エディタ、コンテナ群、LLMのすべてを共存させる必要がある。そのため、メモリ消費の激しいDocker Desktopを廃止し、超軽量でリソース効率の高い「OrbStack」をコンテナランタイムとして採用する。
  - *ローカルLLMの制約*: 常駐によるスワップ発生（SSDの摩耗やシステム全体の遅延）を防ぐため、ローカルLLM（Ollama）は軽量モデル（Qwen2.5-Coder:3b等）に限定し、5分間の非アクティブで自動アンロードする仕様を前提とする。

## 3. アーキテクチャの基本方針とドメイン分離

システムを以下の3つのドメインに分離し、それぞれの責務を明確にする。

### 3.1. プロキシ・ルーティング層 (LiteLLM Proxy)

- **目的**: 外部の商用API（Claude, Gemini等）とローカルLLM（Ollama）を単一のOpenAI互換エンドポイントで抽象化する。
- **背景**: 開発エディタ（Continue）やRAG（Dify）が「現在どのLLMを使っているか」を意識させない疎結合な設計にするため。また、将来的に社内VPCへデプロイした際、この層がAPIキーの隠蔽やコスト監視、プロンプトの無害化（ガードレール）を一元的に担う「関所の役割」を果たすことを実証する。

### 3.2. ナレッジ・RAG層 (Dify)

- **目的**: ドキュメントのアップロード、チャンク分割、ベクトルDB（Weaviate等）への格納、および検索パイプラインをGUIベースで一元管理する。
- **背景**: 自作のPythonスクリプトでRAGを組むことも可能だが、エンタープライズでの運用を見据えた場合、非エンジニアでもナレッジを管理できるDifyのようなプラットフォームの優位性をデモとして提示するため。

### 3.3. エージェント・統合層 (VS Code Continue + MCP)

- **目的**: 開発者がエディタから離れることなく、RAGの知識を引き出しながらコードを自動生成する。
- **背景**: Model Context Protocol (MCP) を用いてContinueとDifyをネイティブに連携させることで、AIに「社内仕様書（ドキュメント）を読んでから実装する」という極めて高度なコンテキスト理解を強制する。

## 4. 高度なPoC機能（非機能要件）

インフラコストを0円（API従量課金のみ）に抑えつつ、エンタープライズの運用に耐えうる以下の自動化フローを組み込む。

- **データ同期の自動化**: Macのローカルフォルダをwatchdogで監視し、ファイル追加時に自動でDifyへインジェストする（S3 Event Notificationsの代替）。
- **ストレージ保護**: ログデータ（SQLite）は7日で自動ローテーションし、限られたローカルSSDの圧迫を防ぐ。
- **自律的修復 (Human-in-the-Loop)**: エージェントがシステム構成の不具合やテスト失敗を検知した場合、自動で修正ブランチを作成し、人間によるマージを待つ安全なCI/CDフローを模倣する。
