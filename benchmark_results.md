# RAGシステム性能調査 (Benchmark Report - REAL SYSTEM)

本システムに導入されたマルチフォーマットパース処理、および Dify ワークフロー連携による Agentic RAG の実機性能測定データです。
※ローカルで起動している Dify API, Redis, および Ollama への実際の通信結果を反映しています。

---

## 1. ドキュメント同期パフォーマンス (パース ＆ 同期所要時間)
ファイルを検知してから、パースを終えて Dify へのアップロードAPIの応答が返るまでの時間です（ネットワーク往復時間を含みます）。

| ドキュメント形式 | 処理時間 (ms) | 特徴と処理内容 |
|---|---|---|
| **Markdown (.md)** | 413.00 ms | テキスト直接アップロード |
| **Word (.docx)** | 84.95 ms | XMLパース ＆ Markdownテーブル変換 ＆ アップロード |
| **Excel (.xlsx)** | 69.99 ms | Pandasテーブル抽出 ＆ markdown形式テーブルアップロード |

### ファイルサイズ・構造による処理時間の変動（スケーラビリティ分析）
上記測定は数キロバイト程度の最小構成テストファイルによる基準値であり、実際のファイルサイズやドキュメントのデータ構造にともなって処理時間は以下のようにスケーリングします。

* **テキスト長・段落数による変動 (docx)**
  Wordファイル内のテキストや表（Table）の走査は要素数 $N$ に対してほぼ線形 $O(N)$ の時間がかかります。特にセル数が多い巨大なテーブルが含まれる場合、Markdown文字列の再構築処理でオーバーヘッドが増加します。
* **シートサイズ・データ量による変動 (xlsx)**
  Excelファイルは Pandas / openpyxl によって一度メモリ上に全展開され、tabulate によって一括で Markdown テーブル文字列へとレンダリングされます。シート数や行数（数万行クラスなど）が肥大化すると、それに伴ってメモリ消費量とレンダリング所要時間が顕著に増加します。
* **スキャン画像・ページ数による変動 (pdf)**
  テキストが存在するPDFは極めて高速に文字抽出できますが、スキャンPDFなど「画像ページ」が存在する場合、1ページごとに **画像化処理 (pdf2image)** で約0.5〜1.5秒、および **Visionモデルによる推論** で約1.0〜3.0秒が累積します。スキャン画像が多いドキュメントの処理時間は、ページ数に対して最も大きく跳ね上がるボトルネックとなります。

## 3. 分析と評価

* **同期時間**: Word や Excel ファイルのパースそのものは数ミリ秒で完了しますが、Dify APIへのドキュメント登録リクエストの往復時間（ネットワークRTTおよびDify内部のDB書込み時間）により、同期全体では数百ミリ秒〜数秒のレイテンシが発生します。これが非同期キューワーカー導入の最大の理由であり、メインスレッドのブロッキングを防ぐ決定打になっています。
* **検索レイテンシ**: Dify の retrieve API を直接叩く通常検索に比べ、Dify ワークフローを実行する Agentic RAG では、ローカル LLM（Qwen2.5-Coder）による推論オーバーヘッドが加わるためレイテンシは大きくなります。しかし、セマンティックキャッシュを有効化することで、同一または極めて類似したクエリは 1ms 前後で返却され、システム全体の総合応答性能を大幅に改善します。
